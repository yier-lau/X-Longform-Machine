# DBS Modules

This skill can use selected dbskill methods as checkpoints. Do not copy the full dbskill knowledge base into the longform machine. The user's personal corpus remains the source of voice and substance.

## Principle

Use dbskill for gates, not for personality.

All dbskill gates require user confirmation before use unless the user explicitly named that dbs skill in the request.

The longform machine owns:

- voice
- diary texture
- personal corpus
- public/private boundary
- X longform output
- publishing package
- cover prompt

dbskill contributes:

- sharper diagnosis
- better first-screen judgment
- AI flavor detection
- concept cleanup
- goal clarity
- state management

## Gate 1: Goal Clarity (`dbs-goal`)

Use when the input is a vague ambition:

- "我想做个人 IP"
- "我想更有影响力"
- "我想表达自己"
- "我想把这些东西做成内容"
- "我想让别人觉得我有价值"

Convert the wish into a checkable article deliverable:

- reader
- changed belief
- core sentence
- proof material
- publishing form
- next asset created

Do not let the article start from empty self-improvement language.

Ask before use:

```text
这里目标还比较空，我建议先过一遍目标清晰化关。确认后我再做。
```

## Gate 2: Concept Deconstruction (`dbs-deconstruct`)

Use when the article depends on fuzzy concepts:

- Agent
- Skill
- Workflow
- MCP
- personal corpus
- creativity
- IP
- influence
- freedom
- system

Output only what the draft needs:

- what the word means here
- what it does not mean
- concrete example
- common misuse
- article-ready definition

Avoid turning this into a philosophy lecture unless the article itself needs that style.

Ask before use:

```text
这个选题里有概念容易糊，我建议先拆一下概念。确认后我再做。
```

## Gate 3: Content Diagnosis (`dbs-content`)

Use after topic, outline, or draft exists.

Check:

- text hygiene
- title/first-screen strength
- expression efficiency
- cognitive gap
- platform fit
- product or asset extension

For X longform, always add:

- Does it sound like the user's lived experience?
- Does it preserve roughness without becoming private oversharing?
- Does it support the personal corpus / longform machine / AI workflow positioning?

Ask before use unless the user requested diagnosis:

```text
现在可以过一遍内容诊断关，主要看开头、表达效率和认知落差。确认后我再做。
```

## Gate 4: Hook And First Screen (`dbs-hook`)

Use when the opening is too slow.

For X longform, the first screen should contain at least one:

- personal scene
- conflict
- anti-common-sense line
- direct pain
- strange but clear phrase

Prefer rewriting the first 5 to 10 lines instead of changing the whole article.

Ask before use:

```text
我觉得第一屏可以再压一下，建议只优化开头，不动全文。确认后我再做。
```

## Gate 5: AI Flavor (`dbs-ai-check`)

Use after a complete draft.

Scan for:

- too smooth
- too complete
- empty parallelism
- generic AI-era slogans
- excessive "we should"
- no lived scene
- no author wound or bias
- no concrete object or action

When the user asks to stop analyzing, do not produce a long diagnostic report. Directly rewrite.

Ask before use unless the user explicitly asks for AI 味检测:

```text
这里可以做一遍 AI 味检查。你要我只诊断，还是直接改？
```

## Gate 6: State Management (`dbs-save`, `dbs-restore`, `dbs-report`)

Use for long-running article projects.

Save or summarize:

- title
- angle
- version history
- best lines
- rejected directions
- privacy notes
- final output paths
- next content ideas

Use state management only when the user asks to save, restore, continue, report, or when a multi-day writing project needs continuity.

Always ask before writing or reading state:

```text
这一步会涉及创作状态存档/恢复/汇总，确认后我再做。
```
